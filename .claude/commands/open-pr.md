Create a pull request that follows `CONTRIBUTING.md`: conventional commit title, **short human-written body**, linked issue, screenshots for UI changes.

## The hard rule this command exists to enforce

From `CONTRIBUTING.md`:

> Long, AI-generated PR descriptions and issues are not acceptable and may be ignored. Respect the maintainers' time: write short, focused descriptions. Explain what changed and why in your own words. If you can't explain it briefly, your PR might be too large.

**This command will not write the PR body for you.** It will ask you for it, in your own words, short. If you paste in AI-generated prose, the command will push back.

## Steps

1. **Verify the working tree** via `git status --short`. If there are uncommitted changes, ask the user: commit them first (which prompts them for a commit message), or stash them, or cancel.

2. **Verify you're not on `main`.** `git branch --show-current`. If you are, stop and instruct the user to create a feature branch first: `git checkout -b feat/<short-slug>` or similar.

3. **Verify `main` is up to date.** `git fetch origin main` (or `upstream main` if working against a fork). If the current branch is behind `main`, warn the user — they may want to rebase or merge before opening the PR.

4. **Run `/verify` first.** If it's not clean, stop and fix before opening the PR. A failing PR wastes maintainer time.

5. **Determine the PR type** from the diff + the linked issue type. This drives the conventional-commit prefix:
   - New feature / new tool / new skill → `feat:` (or `feat(opencode):`)
   - Bug fix → `fix:` (or scoped)
   - Docs only → `docs:`
   - Refactor without behavior change → `refactor:`
   - Tests only → `test:`
   - Perf → `perf:`
   - Maintenance / deps → `chore:`

6. **Ask the user for the linked issue number.** Required. Per Issue-First Policy, no issue = no PR. If they don't have one: stop. Suggest they open an issue first, or ask if the change is so small that a brief issue (one sentence) is appropriate.

7. **Ask the user for the PR title.** Present the conventional-commit prefix you determined in step 5 and ask them to complete it with a short, descriptive title (under 70 characters). Example:
   - Your suggestion: `feat(opencode): `
   - User provides: `add snow_bulk_update_records to operations`
   - Result: `feat(opencode): add snow_bulk_update_records to operations`

8. **Ask the user for a short body.** Three to five sentences, their words. The body should cover:
   - What changed (one sentence)
   - Why (one sentence — link to the issue)
   - How they verified it (one sentence — what did `/verify` run, any manual tests)
   - For UI changes: "Screenshots below" and remind them to attach them after the PR is open
   - For perf changes: include the before/after numbers they ran

   **If the user says "you write it" or "use the diff":** refuse, and explain the "no AI walls of text" rule. Offer to help them phrase it themselves by asking 2–3 short questions instead.

   **If the user pastes in something that reads like AI-generated prose:** push back. Ask them to rewrite it in 2–3 sentences max.

9. **Confirm the plan with the user.** Show:
   - The branch name
   - The PR title (final)
   - The PR body (final)
   - The base branch (should be `main`)
   - `Fixes #<number>` appended to the body

   Wait for explicit "yes" or "go".

10. **Push the branch and create the PR.**
    - `git push -u origin HEAD` (or `upstream HEAD` depending on remotes)
    - `gh pr create --base main --title "<title>" --body "<body>" --draft` — **start as draft** unless the user explicitly asks for a non-draft PR. Draft PRs get maintainer attention when you're ready, and keep the noise down while you're iterating.

11. **Report** the PR URL and a one-line next-step reminder (UI PR? attach screenshots. Perf PR? confirm the numbers are in the body. Any PR? mark as ready for review when you think it's done).

## Safety gates

- **Never `git push --force`.** Never.
- **Never push to `main` directly.** Only to a feature branch.
- **Never skip the Issue-First check.** If there's no issue, stop.
- **Never write the PR body yourself** — ask the user.
- **Never include AI-attribution footers or "generated by" lines** in the commit message or PR body. Maintainers don't want them.
- **If `/verify` failed earlier and the user didn't rerun it after fixing**, refuse to open the PR until a clean `/verify` is confirmed.
- **Start the PR as draft** unless the user explicitly opts out.
